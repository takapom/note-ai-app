# 機能開発ガイド

ドキュメント種別: ガイド。

## フロー

1. 所有者コントラクトを特定します。
2. その機能が MVP コア、MVP support、post-MVP のどれにあたるかを特定します。
3. ポリシーが変わる場合は、実装前にコントラクトを更新します。
4. 検証レーン付きの Superset タスクを作成します。
5. 複雑な変更では Codex を計画優先で実行します。
6. 小さな PR サイズの変更として実装します。
7. トポロジーと SoT/投影の境界に照らしてレビューします。
8. ドキュメントが変わった場合は生成レジスターを更新します。
9. 書き心地、安全性、provenance、reliability、observability に影響する場合は `docs/contracts/non-functional-requirements.md` を確認します。
10. MVP 完了を主張する場合は `docs/contracts/mvp-acceptance.md` を確認します。

## 禁止事項

- 簡単だからという理由で機能を追加してはいけません。
- 所有権を移すために共有パッケージの利便性を利用してはいけません。
- 投影を権威あるものにしてはいけません。
