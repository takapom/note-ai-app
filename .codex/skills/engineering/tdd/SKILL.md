# テスト駆動開発

## 考え方

**中核原則**: テストは実装詳細ではなく、公開インターフェースを通じて振る舞いを検証するべきです。コードは完全に変わってもよい一方で、テストは壊れるべきではありません。

**良いテスト**は統合テスト寄りです。公開 API を通じて実際のコードパスを実行します。システムが _どのように_ 動くかではなく、_何を_ するかを記述します。良いテストは仕様のように読めます。たとえば「有効なカートでユーザーがチェックアウトできる」というテスト名は、どの能力が存在するかを正確に伝えます。この種のテストは内部構造を気にしないため、リファクタリングに耐えます。

**悪いテスト**は実装に結合しています。内部コラボレーターをモックしたり、private メソッドをテストしたり、インターフェースではなく外部手段で検証したりします。たとえばインターフェースを使わず、データベースを直接問い合わせて検証するようなケースです。警告サインは、振る舞いが変わっていないのにリファクタリングでテストが壊れることです。内部関数の名前を変えただけでテストが失敗するなら、そのテストは振る舞いではなく実装をテストしていました。

例は [テスト例](tests.md)、モックの指針は [モックの指針](mocking.md) を参照してください。

## アンチパターン: 水平スライス

**最初にすべてのテストを書き、その後ですべての実装を書く、という進め方はしないでください。** これは「水平スライス」です。RED を「すべてのテストを書く」、GREEN を「すべてのコードを書く」と扱っています。

これは **粗悪なテスト** を生みます。

- まとめて書いたテストは _実際の_ 振る舞いではなく、_想像上の_ 振る舞いをテストする
- ユーザーから見える振る舞いではなく、ものの _形_、つまりデータ構造や関数シグネチャをテストしがちになる
- テストが実際の変化に鈍くなる。振る舞いが壊れても通り、振る舞いが正しくても失敗する
- 見通せる範囲を超えて進み、実装理解の前にテスト構造へコミットしてしまう

**正しい進め方**: トレーサーバレットによる垂直スライス。1 つのテスト → 1 つの実装 → 繰り返し。各テストは、前のサイクルで学んだことに応じて書きます。書いたばかりのコードなので、どの振る舞いが重要で、どう検証すべきかを正確に把握できます。

```
悪い例 (水平):
  RED:   test1, test2, test3, test4, test5
  GREEN: impl1, impl2, impl3, impl4, impl5

良い例 (垂直):
  RED→GREEN: test1→impl1
  RED→GREEN: test2→impl2
  RED→GREEN: test3→impl3
  ...
```

## ワークフロー

### 1. 計画

コードベースを探索するときは、テスト名とインターフェースの語彙がプロジェクトの言葉に合うよう、プロジェクトのドメイン用語集を使います。触る領域の ADR も尊重します。

コードを書く前に:

- [ ] どのインターフェース変更が必要かをユーザーに確認する
- [ ] どの振る舞いをテストするかをユーザーに確認する。優先順位を付ける
- [ ] [深いモジュール](deep-modules.md)、つまり小さなインターフェースと深い実装の機会を見つける
- [ ] [テストしやすさ](interface-design.md) を考慮してインターフェースを設計する
- [ ] テストする振る舞いを列挙する。実装手順ではなく振る舞いを列挙する
- [ ] 計画についてユーザーの承認を得る

尋ねること: 「公開インターフェースはどのような形にすべきですか？ どの振る舞いを最優先でテストすべきですか？」

**すべてをテストすることはできません。** どの振る舞いが最も重要かをユーザーと正確に確認します。すべての境界ケースではなく、重要経路と複雑なロジックにテスト労力を集中させます。

### 2. トレーサーバレット

システムについて 1 つのことだけを確認するテストを 1 つ書きます。

```
RED:   最初の振る舞いのテストを書く → テストが失敗する
GREEN: 通すための最小コードを書く → テストが通る
```

これがトレーサーバレットです。経路が端から端まで機能することを証明します。

### 3. インクリメンタルループ

残りの各振る舞いについて:

```
RED:   次のテストを書く → 失敗する
GREEN: 通すための最小コードを書く → 通る
```

ルール:

- 一度に 1 つのテスト
- 現在のテストを通すために必要な分だけコードを書く
- 将来のテストを先回りしない
- 観測可能な振る舞いにテストを集中させる

### 4. リファクタリング

すべてのテストが通った後、[リファクタリング候補](refactoring.md) を探します。

- [ ] 重複を抽出する
- [ ] モジュールを深くする。単純なインターフェースの背後へ複雑さを移す
- [ ] 自然に当てはまる場合は SOLID 原則を適用する
- [ ] 新しいコードが既存コードについて何を明らかにしたかを考える
- [ ] 各リファクタリング手順の後にテストを実行する

**RED の間は絶対にリファクタリングしないでください。** まず GREEN にします。

## サイクルごとのチェックリスト

```
[ ] テストは実装ではなく振る舞いを記述している
[ ] テストは公開インターフェースだけを使っている
[ ] テストは内部リファクタリングに耐える
[ ] コードはこのテストに対して最小限である
[ ] 推測による機能追加をしていない
```
