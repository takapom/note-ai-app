# インターフェース設計

ユーザーが、選ばれた深化候補について代替インターフェースを探索したい場合は、この並列 sub-agent pattern を使う。"Design It Twice" (Ousterhout) に基づく。最初の案が最善である可能性は低い。

[LANGUAGE.md](LANGUAGE.md) の語彙を使う。つまり **モジュール**、**インターフェース**、**継ぎ目**、**アダプター**、**レバレッジ**。

## プロセス

### 1. 問題空間を定義する

sub-agent を起動する前に、選ばれた候補の問題空間をユーザー向けに説明する。

- 新しいインターフェースが満たす必要のある制約
- 依存するものと、それぞれがどのカテゴリに入るか ([DEEPENING.md](DEEPENING.md) を参照)
- 制約に足場をかけるための、おおまかな説明用コードスケッチ。提案ではなく、制約を具体化するためのもの

これをユーザーに見せたら、すぐに Step 2 へ進む。ユーザーは sub-agent が並列で作業している間に読み、考える。

### 2. sub-agent を起動する

Agent ツールを使い、3 つ以上の sub-agent を並列起動する。それぞれが、深めたモジュールの**根本的に異なる**インターフェースを作ること。

各 sub-agent には別々の技術 brief を渡す。内容は、file path、結合の詳細、[DEEPENING.md](DEEPENING.md) 由来の依存カテゴリ、継ぎ目の背後にあるもの。brief は、Step 1 のユーザー向け問題空間説明とは独立させる。各 agent に異なる設計制約を与える。

- Agent 1: "インターフェースを最小化する。入口は最大 1-3 個を目指す。入口 1 つあたりのレバレッジを最大化する。"
- Agent 2: "柔軟性を最大化する。多くの use case と extension を支える。"
- Agent 3: "最も一般的な caller に最適化する。default case を trivial にする。"
- Agent 4 (該当する場合): "継ぎ目を越える依存のために、ports & adapters を中心に設計する。"

各 sub-agent がアーキテクチャ言語とプロジェクトのドメイン言語を一貫して使えるよう、brief には [LANGUAGE.md](LANGUAGE.md) の語彙と CONTEXT.md の語彙の両方を含める。

各 sub-agent は次を出力する。

1. インターフェース。types、methods、params に加え、不変条件、順序、エラーモードを含む
2. caller がどう使うかを示す usage example
3. 実装が継ぎ目の背後に隠すもの
4. 依存戦略とアダプター ([DEEPENING.md](DEEPENING.md) を参照)
5. trade-off。どこでレバレッジが高く、どこが薄いか

### 3. 提示して比較する

ユーザーが各案を吸収できるよう、design を順番に提示する。その後、文章で比較する。**深さ**、つまりインターフェース上のレバレッジ、**局所性**、つまり変更がどこに集中するか、そして**継ぎ目の配置**で対比する。

比較したら、自分の推奨を出す。どの design が最も強いと思うか、なぜそう思うかを述べる。異なる design の要素を組み合わせると良い場合は、hybrid を提案する。意見を持つこと。ユーザーはメニューではなく、強い読みを求めている。
