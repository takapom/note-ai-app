# 深化

浅いモジュール群を、その依存関係を踏まえて安全に深める方法。[LANGUAGE.md](LANGUAGE.md) の語彙を前提にする。つまり **モジュール**、**インターフェース**、**継ぎ目**、**アダプター**。

## 依存カテゴリ

深化候補を評価するときは、その依存を分類する。カテゴリによって、深めたモジュールを継ぎ目越しにどうテストするかが決まる。

### 1. In-process

pure computation、in-memory state、I/O なし。常に深化できる。モジュールを統合し、新しいインターフェース越しに直接テストする。アダプターは不要。

### 2. Local-substitutable

ローカルのテスト用代替物を持つ依存。例: Postgres に対する PGLite、in-memory filesystem。代替物が存在するなら深化できる。深めたモジュールは、テストスイート内で代替物を動かしてテストする。継ぎ目は内部に留める。モジュールの外部インターフェースに port を出さない。

### 3. Remote but owned (Ports & Adapters)

ネットワーク越しにある自分たち所有のサービス。例: microservices、internal APIs。継ぎ目に **port**、つまりインターフェースを定義する。深いモジュールはロジックを所有し、transport は **アダプター**として注入する。テストは in-memory アダプターを使う。本番は HTTP/gRPC/queue アダプターを使う。

推奨の形: _"継ぎ目に port を定義し、本番用に HTTP アダプター、テスト用に in-memory アダプターを実装する。そうすればネットワーク越しにデプロイされていても、ロジックは 1 つの深いモジュールに置ける。"_

### 4. True external (Mock)

自分たちで制御できない third-party service。例: Stripe、Twilio など。深めたモジュールは外部依存を注入された port として受け取り、テストでは mock アダプターを提供する。

## 継ぎ目の規律

- **アダプター 1 つは仮説上の継ぎ目。アダプター 2 つは実在する継ぎ目。** 少なくとも 2 つのアダプターが正当化されない限り、port を導入しない。典型的には本番 + テストである。アダプターが 1 つだけの継ぎ目は、単なる間接参照でしかない。
- **内部の継ぎ目と外部の継ぎ目を分ける。** 深いモジュールは、インターフェース上の外部の継ぎ目に加えて、内部の継ぎ目を持てる。内部の継ぎ目は実装に閉じ、自身のテストで使うもの。テストが使うからといって、内部の継ぎ目をインターフェースへ公開しない。

## テスト戦略: 積み重ねず、置き換える

- 浅いモジュールの古い unit test は、深めたモジュールのインターフェースに対するテストができた時点で無駄になる。削除する。
- 深めたモジュールのインターフェースに対して新しいテストを書く。**インターフェースがテスト面である。**
- テストは内部状態ではなく、インターフェース越しに観測できる結果を assert する。
- テストは内部リファクタリングに耐えるべきである。テストは実装ではなく振る舞いを記述する。実装が変わったときにテストも変える必要があるなら、そのテストはインターフェースの先を覗いている。
