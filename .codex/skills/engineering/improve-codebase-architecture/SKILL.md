# コードベースアーキテクチャの改善

アーキテクチャ上の摩擦を表面化し、**深化機会**を提案する。深化機会とは、浅いモジュールを深いモジュールへ変えるリファクタリングである。狙いは、テストしやすさと AI が辿りやすい構造である。

## 用語集

すべての提案で、次の用語を厳密に使う。一貫した言葉遣いが重要なので、「コンポーネント」「サービス」「API」「境界」へ流れないこと。完全な定義は [LANGUAGE.md](LANGUAGE.md) にある。

- **モジュール** — インターフェースと実装を持つものすべて。関数、クラス、パッケージ、スライスを含む。
- **インターフェース** — caller がモジュールを使うために知る必要があるものすべて。型、不変条件、エラーモード、順序、設定を含む。型シグネチャだけではない。
- **実装** — 内側のコード。
- **深さ** — インターフェース上のレバレッジ。小さなインターフェースの背後に多くの振る舞いがある状態。**深い** = 高レバレッジ。**浅い** = インターフェースが実装とほぼ同じくらい複雑。
- **継ぎ目** — インターフェースが存在する場所。その場を直接編集せずに振る舞いを変えられる場所。(「境界」ではなく、この語を使う。)
- **アダプター** — 継ぎ目でインターフェースを満たす具体物。
- **レバレッジ** — 深さによって caller が得るもの。
- **局所性** — 深さによって maintainer が得るもの。変更、バグ、知識が一箇所に集中する。

主要原則。完全な一覧は [LANGUAGE.md](LANGUAGE.md) を参照。

- **削除テスト**: そのモジュールを削除することを想像する。複雑さが消えるなら、それは単なる通過点だった。複雑さが N 個の caller に再出現するなら、そのモジュールは役割を果たしていた。
- **インターフェースがテスト面である。**
- **アダプター 1 つ = 仮説上の継ぎ目。アダプター 2 つ = 実在する継ぎ目。**

このスキルは、プロジェクトのドメインモデルから情報を得る。ドメイン語彙は良い継ぎ目に名前を与える。ADR は、このスキルが蒸し返すべきでない判断を記録している。

## プロセス

### 1. 探索

まず、プロジェクトのドメイン用語集と、触る領域に関係する ADR を読む。

次に Agent ツールを `subagent_type=Explore` で使い、コードベースを歩く。硬直したヒューリスティックには従わない。自然に探索し、自分が摩擦を感じた場所を記録する。

- 1 つの概念を理解するために、多数の小さなモジュールを行き来する必要がある場所はどこか。
- **浅い**モジュールはどこか。つまり、インターフェースが実装とほぼ同じくらい複雑な場所はどこか。
- テストしやすさのためだけに pure function が切り出されているが、本当のバグはそれらの呼び出し方に隠れている場所はどこか。つまり**局所性**がない場所はどこか。
- 密結合したモジュールが、継ぎ目を越えて漏れている場所はどこか。
- コードベースのどの部分が未テストか、または現在のインターフェース越しにはテストしにくいか。

浅いと疑うものには**削除テスト**を適用する。削除したら複雑さは集中するのか、それとも単に移動するだけか。「集中する」が欲しいシグナルである。

### 2. 候補を HTML レポートとして提示する

自己完結した HTML ファイルを OS の一時ディレクトリへ書き出し、repo には何も残さない。一時ディレクトリは `$TMPDIR` から解決し、なければ `/tmp`、Windows では `%TEMP%` にフォールバックする。各実行で新しいファイルになるよう `<tmpdir>/architecture-review-<timestamp>.html` へ書く。ユーザー向けに開く。Linux では `xdg-open <path>`、macOS では `open <path>`、Windows では `start <path>` を使い、絶対パスを伝える。

レポートはレイアウトとスタイルに **Tailwind via CDN** を使い、グラフ、フロー、シーケンスが構造を確実に伝える場合は **Mermaid via CDN** を使う。Mermaid と手作りの CSS/SVG ビジュアルを混ぜる。関係がグラフ状なら Mermaid、より編集的な表現が必要なら手作りの div/SVG を使う。例: mass diagram、cross-section、collapse animation。各候補には **before/after の可視化**を付ける。視覚で伝える。

各候補は以前と同じテンプレートだが、カードとして描画する。

- **Files** — 関係するファイルやモジュール
- **Problem** — 現在のアーキテクチャが摩擦を生んでいる理由
- **Solution** — 何を変えるかの平易な説明
- **Benefits** — 局所性とレバレッジ、およびテストの改善として説明する
- **Before / After diagram** — 浅さと深化を示す、左右並びのカスタム図
- **Recommendation strength** — `Strong`、`Worth exploring`、`Speculative` のいずれか。badge として描画する

レポートの最後に **Top recommendation** セクションを置く。最初に取り組むべき候補と、その理由を書く。

**ドメインには CONTEXT.md の語彙を使い、アーキテクチャには [LANGUAGE.md](LANGUAGE.md) の語彙を使う。** `CONTEXT.md` が "Order" を定義しているなら、"Order intake module" と呼ぶ。"FooBarHandler" や "Order service" とは呼ばない。

**ADR との衝突**: 候補が既存 ADR と矛盾する場合、本当に ADR を見直すだけの摩擦があるときだけ表面化する。カード内で明確に示す。例: _"contradicts ADR-0007 — but worth reopening because..."_。ADR が禁じている理論上のリファクタリングをすべて列挙しない。

HTML の完全な scaffold、diagram pattern、styling guidance は [HTML-REPORT.md](HTML-REPORT.md) を参照。

この時点では、まだインターフェースを提案しない。ファイルを書き出したら、ユーザーにこう尋ねる: "この中で、どれを掘り下げたいですか？"

### 3. 詰める会話ループ

ユーザーが候補を選んだら、詰める会話に入る。制約、依存、深めたモジュールの形、継ぎ目の背後に置くもの、生き残るテストを、設計ツリーとして一緒に辿る。

判断が固まり次第、副作用はその場で発生させる。

- **`CONTEXT.md` にない概念で、深めたモジュールに名前を付ける場合**: `/grill-with-docs` と同じ規律で、その用語を `CONTEXT.md` に追加する ([CONTEXT-FORMAT.md](../grill-with-docs/CONTEXT-FORMAT.md) を参照)。ファイルがなければ遅延作成する。
- **会話中に曖昧な用語が鋭くなった場合**: その場で `CONTEXT.md` を更新する。
- **ユーザーが、構造上重要な理由で候補を却下した場合**: ADR を提案する。言い方はこうする: _"この判断を ADR として記録し、今後のアーキテクチャレビューで同じ提案が再提示されないようにしますか？"_ 将来の探索者が同じ提案を避けるために本当に必要な理由に限る。"今は割に合わない" のような一時的理由や、自明な理由では提案しない。[ADR-FORMAT.md](../grill-with-docs/ADR-FORMAT.md) を参照。
- **深めたモジュールの代替インターフェースを探索したい場合**: [INTERFACE-DESIGN.md](INTERFACE-DESIGN.md) を参照。
