# Cross Aggregate Constraints

集約間制約は、すぐ実装方法に飛ばず、まず要件とモデリングを疑う。

## 判断フロー

1. その制約は本当に強い一貫性が必要か確認する。
2. 強い一貫性が必要なら、同一集約にすべき不変条件ではないか確認する。
3. 別集約のままなら、制約を緩められるか、状態遷移で表現できるかを検討する。
4. 結果整合性でよいなら、イベント、ポリシー、補償、警告、後続処理で扱う。
5. CQRS/ES の場合、コマンド判断に read model を使う危険を明示する。

## 設計規律

- コマンド側の判断は、原則としてコマンド側の状態で完結させる。
- read model は投影であり、ドメイン判断の source of truth にしない。
- Saga は長いプロセスと補償のための道具であり、同期的な制約チェックの代用品ではない。
- 他集約の存在確認は、必要ならアプリケーションサービスの事前条件として扱い、ドメイン不変条件と混同しない。

## アンチパターン

- 削除可否を判断するために、コマンド側から巨大な read model を参照する。
- 集約 A のメソッド内で集約 B のリポジトリを呼ぶ。
- Saga で「他集約が条件を満たすか」を同期的に確認しようとする。
- 不整合を許容できない要件なのに、別集約に分けたままにする。

## レビューチェックリスト

- その制約は本当に即時一貫性が必要か。
- 制約の所有者はどの集約か説明できるか。
- 別集約の状態を直接読まずに設計できるか。
- 不整合が起きた場合の許容、検知、補償が定義されているか。
- read model を source of truth として扱っていないか。
