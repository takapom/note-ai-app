# Aggregate Design

集約を「オブジェクトのまとまり」ではなく、真の不変条件を守る強い整合性境界として扱う。

## 判断フロー

1. その概念に識別子、ライフサイクル、永続化境界、真の不変条件があるか確認する。なければ集約にしない。
2. コマンド直後に必ず真であるべきルールを列挙する。
3. そのルールを守るために同時変更が必要な最小オブジェクトだけを集約内に入れる。
4. 他集約は実体参照せず ID 参照にする。
5. 集約外の整合性はドメインイベント、ポリシー、Saga、結果整合性で扱う。

## 設計規律

- 集約ルートだけを外部からの入口にする。
- 子エンティティは集約内のローカルなライフサイクルに閉じる。
- public field、可変 collection、外部からの append/push を禁止する。
- 状態変更は不変条件を通る intention-revealing method に閉じる。
- 集約は小さく保つ。UI で一緒に見えることや DB 外部キーだけで含めない。
- 可能なら immutable update を優先する。可変にする場合も不変条件を破る経路を閉じる。

## アンチパターン

- `Order.customer: Customer` のように別集約を直接持つ。
- `order.items.push(item)` のように子要素を外部から変更できる。
- `Product` 集約が大量の `BacklogItem` や `Comment` を抱える。
- 集約が保存方法、ORM、HTTP、トランザクション API を知っている。
- 集約にする根拠が「テーブルがある」「画面に表示する」だけである。

## レビューチェックリスト

- 集約境界は真の不変条件で説明できるか。
- 1コマンドで更新する範囲は1集約に閉じているか。
- 他集約は ID のみで参照しているか。
- 集約ルート以外から内部エンティティを直接変更できないか。
- 不正な状態をコンストラクタ、factory、更新メソッドで拒否しているか。
- 集約外の整合性を同期的な直接更新にしていないか。
