# Domain Building Blocks

DDD の building block は分類名ではなく、責務の置き場所を決めるために使う。

## 判断フロー

1. 値だけで等価性が決まり、不変条件や計算を持つなら値オブジェクトにする。
2. アイデンティティとライフサイクルが本質ならエンティティにする。
3. 複数オブジェクトの不変条件を守る境界が必要なら集約にする。
4. どの値オブジェクトや集約にも自然に属さないドメイン判断だけをドメインサービスにする。
5. 調整、永続化、外部 API 呼び出しはアプリケーションサービスに置く。

## 配置規律

- 値オブジェクトは常に valid な状態で生成する。
- エンティティは自分のライフサイクルを守るが、所属集約の不変条件を破らない。
- 集約ルートは外部からの変更を集約する。
- ドメインサービスは例外的に使う。名前は技術動詞ではなくドメイン上の判断にする。

## アンチパターン

- すべてを `Service` に寄せる。
- ID がないのにエンティティにする。
- ライフサイクルがある値を値オブジェクトにする。
- 値オブジェクトに setter や永続 ID を持たせる。
- ドメインサービスがリポジトリを大量に受け取り、手続き型モデルになる。

## レビューチェックリスト

- その型の等価性は値か ID か。
- その状態の所有者は誰か。
- 不変条件は型、エンティティ、集約のどこで閉じているか。
- ドメインサービスに置いた処理は、本当に単一モデルへ置けないか。
- アプリケーションサービスにドメイン判断が漏れていないか。
